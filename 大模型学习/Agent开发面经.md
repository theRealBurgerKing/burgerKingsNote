## 讲讲你对agent的理解
一个经典的 Agent 运行循环是 **Reason → Act → Observe**，也就是 ReAct 范式：

```
目标输入 → LLM推理reasoning → 调用工具acting → 获取结果observe → 再次推理 → ... → 输出最终答案
```


## 讲讲你知道的框架，cot tot react等
### 1. CoT（Chain of Thought）—— 思维链

CoT 是让模型在给出最终答案之前，==**逐步输出中间推理过程**==，模拟人类"一步一步思考"的方式。

**核心思想：** prompt 里加入 _"Let's think step by step"_ 或提供 few-shot 示例，引导模型显式展开推理链。

**局限：** ==推理链是线性的，一旦某步出错，后续全错，且没有自我纠错能力。==
### 2. ToT（Tree of Thoughts）—— 思维树

ToT 是 CoT 的扩展，把线性的推理链升级为==**树状==搜索结构**。

**核心思想：**

- 每一步生成**多个候选思维节点**（branching）
- 通过==评估函数==对节点打分
- ==搜索算法（BFS/DFS）==
- ==支持**回溯**==（backtracking），走错了可以退回重来


### 为什么需要 ToT？

CoT 的本质是**贪心搜索**——每一步只生成一个 token/思维，一旦走偏就无法回头。

人类解决难题时并不是这样的，我们会：

- 同时考虑多个方案
- 评估哪条路更有希望
- 走不通了就退回来换一条

ToT 就是把这种**人类启发式搜索**的思维方式赋予 LLM。



### 3. ReAct（Reasoning + Acting）

ReAct 是 Agent 开发中最经典的框架，将**推理（Reasoning）和行动（Acting）交织在一起**，形成一个循环。

**核心循环：**

```
Thought → Action → Observation → Thought → Action → ...
```

- **Thought**：模型内部推理，分析当前状态，决定下一步
- **Action**：调用外部工具（搜索、代码执行、API等）
- **Observation**：获取工具返回结果，注入上下文

**优势：** 推理过程透明可解释，工具调用有依据，错误可追踪

**局限：** 遇到复杂任务时，Thought 质量高度依赖模型能力，且没有显式的规划阶段

**代表实现：** LangChain 的 ReAct Agent、早期的 AutoGPT

---

### 4. 其他值得一提的框架

**Plan-and-Execute（规划-执行分离）**

- ==先由 Planner 生成完整任务计划==
- ==再由 Executor 逐步执行==
- 比 ReAct 更适合长流程、多步骤任务

进阶：Re-Planning 机制

静态计划有个问题——执行到一半发现计划不可行怎么办？

实际落地会加入 **Re-Planning**：


**Reflexion**

- 在 ReAct 基础上加入==**自我反思**==机制
- Agent 执行失败后，生成语言反馈存入记忆，下次避免同样错误

**Self-Consistency**

- CoT 的增强版，==采样多条推理路径，投票选最一致的答案。==




## Agent 的长短期记忆怎么做

### 短期记忆

本质就是 ==**Context Window**，放在 prompt== 里的所有内容：

```
System Prompt
    + 历史对话（滑动窗口）
    + 当前工具调用结果
    + 当前思维链
─────────────────────
= 短期记忆的全部
```

**管理策略：**

|策略|做法|适用|
|---|---|---|
|滑动窗口|只保留最近 N 轮对话|通用对话|
|摘要压缩|旧对话让 LLM 摘要后存入|长对话|
|重要性过滤|只保留关键信息节点|任务型 Agent|

---

### 长期记忆

需要**外部存储**，核心是两种：

#### ① Episodic Memory（片段记忆）

存储历史完整事件，类似日记：

```
"2024-01-01: 用户让我订机票，我用了相对日期导致失败，
 应该先把'明天'转成具体日期"
```

实现方式：数据库直接存文本，按时间或相关性检索

#### ② Semantic Memory（语义记忆）

存储结构化知识和事实，用**向量数据库**实现：

```
用户偏好、领域知识、工具使用经验
─────────────────────────────────
Embedding → 向量库（Pinecone/Chroma/Milvus）
检索时：query embedding → 相似度搜索 → top-k 召回
```

#### ③ Procedural Memory（程序记忆）

存储"怎么做"的技能，体现为：

- System Prompt 里的固定指令
- Few-shot 示例
- 工具的使用规范

---

### 记忆写入与读取的完整流程

```
用户输入
   ↓
[记忆读取] 
   → 向量检索相关长期记忆并拼入短期上下文
   ↓
[Agent 执行]
   ↓
[记忆写入]
   → 更新长期记忆
   → 更新短期滑动窗口
```

---

## 二、长短期记忆引出的问题

### 问题1：==Context Window 长度限制==

短期记忆装不下怎么办？

**方案：**

- ==**摘要压缩**：让 LLM 把旧对话总结成几句话==
- ==**RAG 化**：把历史对话也向量化，按需检索而不是全部塞入==

---

### 问题2：==记忆的噪声与幻觉==

记忆本身是 LLM 提取和生成的，天然携带幻觉风险

**方案：**

- 记忆写入时加置信度评分
- 定期做记忆审计和去重
- 检索结果加来源标注，让模型自己判断可信度
区分"事实"和"推断"，打上标签。→ 推断类记忆不直接使用，只作参考 → 高重要性记忆使用前让 LLM 再验证一次

---

### 问题3：==记忆检索的相关性问题==

==向量相似度≠语义真正相关==，可能检索回来的记忆根本没用，反而干扰推理。

**方案：**

- **混合检索**：向量检索 + BM25 关键词检索结合
向量检索强在语义理解，BM25（TFIDF升级版）强在关键词匹配
- **Re-ranking**：用 Cross-Encoder 对召回结果重排序
- **检索后过滤**：让 LLM 判断召回记忆是否真正相关

---

### 问题4：==记忆的一致性问题==

用户的偏好、状态会随时间变化，旧记忆可能已经过时。

```
记忆："用户喜欢Python"  ← 三个月前存的
现实：用户已经转Go了
```

**方案：**

- 记忆存入时打时间戳
- 检索时按时间衰减加权
- 新信息和旧记忆冲突时，优先新信息并更新记忆

---

### 问题5：记忆的隐私与安全

多用户场景下，记忆隔离必须严格，否则会有数据泄露风险。

**方案：**

- 记忆按用户 ID 严格隔离
- 敏感信息加密存储
- 访问控制 + 审计日志

---

### 问题6：记忆爆炸（Memory Bloat）

长期运行后记忆库越来越大，检索变慢，噪声增多。
”噪声比信号增长更快“
**方案：**
阶段一：写入时就控制质量（预防）
这是最根本的解法，**垃圾不入库**比入库后再清理效率高得多。

去重检测

信息提炼写入（不存原文）

阶段二：
聚类合并
对于已有记忆进行定期的整理，把相似的记忆合并

阶段三：
遗忘
不同类型记忆的半衰期配置
时效性+重要性+访问频率+置信度 加权综合


冷热分层
┌─────────────────────────────────────────────────────┐
│  🔥 热层 Hot Layer                                   │
│                                                     │
│  存储：内存向量索引（FAISS / Milvus 内存模式）        │
│  容量：每用户 ≤ 3000 条                              │
│  检索：< 30ms                                       │
│  内容：近90天活跃记忆 + 核心偏好 + 永久记忆           │
├─────────────────────────────────────────────────────┤
│  🌡️ 温层 Warm Layer                                  │
│                                                     │
│  存储：磁盘向量索引（Milvus / Weaviate）              │
│  容量：每用户 ≤ 20000 条                             │
│  检索：50~200ms                                     │
│  内容：90天~1年的记忆 + 摘要层数据                   │
├─────────────────────────────────────────────────────┤
│  ❄️ 冷层 Cold Layer                                  │
│                                                     │
│  存储：对象存储（S3 / OSS）+ 简单倒排索引             │
│  容量：无限                                          │

## 三、Agent 任务分解怎么实现

### 为什么需要任务分解

复杂任务直接扔给 Agent 会：

- 超出单次推理能力边界
- 无法并行，效率低
- 出错时难以定位和恢复

任务分解的目标是把一个**大而模糊的目标**变成**小而具体的可执行步骤**。

---

### 方法一：LLM 直接分解（最常用）

直接 prompt 让模型输出结构化任务树：

```
Prompt:
"你是一个任务规划专家。请将以下任务分解为
可独立执行的子任务，输出 JSON 格式：

任务：'帮我分析竞品并生成报告'

输出格式：
{
  'tasks': [
    {'id': 1, 'desc': '...', 'depends_on': []},
    {'id': 2, 'desc': '...', 'depends_on': [1]},
    ...
  ]
}"
```

输出示例：

json

```json
{
  "tasks": [
    {"id": 1, "desc": "确定竞品列表", "depends_on": []},
    {"id": 2, "desc": "抓取竞品官网信息", "depends_on": [1]},
    {"id": 3, "desc": "分析各竞品核心功能", "depends_on": [2]},
    {"id": 4, "desc": "生成对比分析报告", "depends_on": [3]}
  ]
}
```

---

### 方法二：层级分解（Hierarchical）

对于特别复杂的任务，做**多层分解**：

```
总目标：开发一个电商网站
   ├── 子任务1：前端开发
   │     ├── 子子任务1.1：搭建页面框架
   │     ├── 子子任务1.2：实现商品列表
   │     └── 子子任务1.3：实现购物车
   ├── 子任务2：后端开发
   │     ├── 子子任务2.1：设计数据库
   │     └── 子子任务2.2：实现API
   └── 子任务3：测试部署
```

实现上可以**递归调用分解 prompt**，直到每个叶子节点足够原子化。

---

### 方法三：依赖图 + 并行执行

分解完任务后，识别依赖关系，**无依赖的任务并行执行**：

```
Task1 ──────────────────→ Task4
Task2 ──┐                  ↑
        ├──→ Task3 ────────┘
Task2 ──┘

Task1 和 Task2 无依赖 → 并行执行
Task3 依赖 Task1、Task2 → 等待后执行
Task4 依赖 Task3 → 最后执行
```

工程实现用 **DAG（有向无环图）**：

python

```python
# 伪代码
dag = build_dag(tasks)  # 构建依赖图
ready_queue = dag.get_ready_tasks()  # 获取无依赖任务

while not dag.is_complete():
    parallel_execute(ready_queue)  # 并行执行
    dag.mark_complete(done_tasks)
    ready_queue = dag.get_ready_tasks()  # 解锁新任务
```

---

### 方法四：动态分解（边执行边分解）

不预先生成完整计划，而是**执行一步再决定下一步**，更灵活：

```
目标 → 分解出 Step1
         ↓ 执行 Step1
       根据结果分解 Step2
         ↓ 执行 Step2
       根据结果分解 Step3
         ↓ ...
```

适合：任务高度动态，前一步的结果决定后续走向。

这本质上就是 **ReAct 的 Thought 层做任务分解**。





## Embedding的原理
把文本（或图片、音频等）转换成高维空间中的一个**数值向量**，语义相近的内容在这个空间里距离也近。检索时，把 query 也转成向量，找最近邻即可。

**余弦相似度**cos(θ) = A·B / (‖A‖‖B‖)最常用，忽略向量长度

ANN 索引算法（为什么不直接暴力搜）
百万级向量暴力计算太慢，实际用近似最近邻（ANN）
