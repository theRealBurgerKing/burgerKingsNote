## 讲讲你知道的框架，cot tot react等
### 1. CoT（Chain of Thought）—— 思维链

CoT 是让模型在给出最终答案之前，==**逐步输出中间推理过程**==，模拟人类"一步一步思考"的方式。

**核心思想：** prompt 里加入 _"Let's think step by step"_ 或提供 few-shot 示例，引导模型显式展开推理链。

**局限：** ==推理链是线性的，一旦某步出错，后续全错，且没有自我纠错能力。==
### 2. ToT（Tree of Thoughts）—— 思维树

ToT 是 CoT 的扩展，把线性的推理链升级为==**树状==搜索结构**。

**核心思想：**

- 每一步生成**多个候选思维节点**（branching）
- 通过==评估函数==对节点打分
- ==搜索算法（BFS/DFS）==
- ==支持**回溯**==（backtracking），走错了可以退回重来


### 为什么需要 ToT？

CoT 的本质是**贪心搜索**——每一步只生成一个 token/思维，一旦走偏就无法回头。

人类解决难题时并不是这样的，我们会：

- 同时考虑多个方案
- 评估哪条路更有希望
- 走不通了就退回来换一条

ToT 就是把这种**人类启发式搜索**的思维方式赋予 LLM。



### 3. ReAct（Reasoning + Acting）

ReAct 是 Agent 开发中最经典的框架，将**推理（Reasoning）和行动（Acting）交织在一起**，形成一个循环。

**核心循环：**

```
Thought → Action → Observation → Thought → Action → ...
```

- **Thought**：模型内部推理，分析当前状态，决定下一步
- **Action**：调用外部工具（搜索、代码执行、API等）
- **Observation**：获取工具返回结果，注入上下文

**优势：** 推理过程透明可解释，工具调用有依据，错误可追踪

**局限：** 遇到复杂任务时，Thought 质量高度依赖模型能力，且没有显式的规划阶段

**代表实现：** LangChain 的 ReAct Agent、早期的 AutoGPT

---

### 4. 其他值得一提的框架

**Plan-and-Execute（规划-执行分离）**

- ==先由 Planner 生成完整任务计划==
- ==再由 Executor 逐步执行==
- 比 ReAct 更适合长流程、多步骤任务

进阶：Re-Planning 机制

静态计划有个问题——执行到一半发现计划不可行怎么办？

实际落地会加入 **Re-Planning**：


**Reflexion**

- 在 ReAct 基础上加入==**自我反思**==机制
- Agent 执行失败后，生成语言反馈存入记忆，下次避免同样错误

**Self-Consistency**

- CoT 的增强版，==采样多条推理路径，投票选最一致的答案。==

### 短期记忆

本质就是 ==**Context Window**，放在 prompt== 里的所有内容：

### 方式二：滑动窗口

只保留最近 N 轮：
简单够用，但早期信息会丢失。

实现：队列

---

### 方式三：摘要压缩

历史过长时，让模型先总结，用摘要替代原始历史：

```
原始10轮对话
     ↓ 超过阈值时
让LLM总结成："用户是Allen，正在做NLP项目，询问过embedding和function calling相关问题"
     ↓
摘要 + 最近3轮  → 作为新的 context
```

LangChain 的 `ConversationSummaryBufferMemory` 就是这个思路。

---

### 方式四：向量记忆（长期记忆）

把历史对话存成向量，每次检索相关片段注入 context，适合**超长期记忆**：

```
历史对话 → embedding → 存入向量库

新消息来了：
query → 检索最相关的历史片段 → 拼入 context → 请求模型
```

==实际项目里一般是**滑动窗口 + 摘要**组合，窗口内保留原始对话，窗口外的压缩成摘要追加到 system prompt 里。==




---

### 长期记忆

需要**外部存储**，核心是两种：

#### ① Episodic Memory（片段记忆）

存储历史完整事件，类似日记：

```
"2024-01-01: 用户让我订机票，我用了相对日期导致失败，
 应该先把'明天'转成具体日期"
```

实现方式：数据库直接存文本，按时间或相关性检索

#### ② Semantic Memory（语义记忆）

存储结构化知识和事实，用**向量数据库**实现：

```
用户偏好、领域知识、工具使用经验
─────────────────────────────────
Embedding → 向量库（Pinecone/Chroma/Milvus）
检索时：query embedding → 相似度搜索 → top-k 召回
```

#### ③ Procedural Memory（程序记忆）

存储"怎么做"的技能，体现为：

- System Prompt 里的固定指令
- Few-shot 示例
- 工具的使用规范

---

## 二、长短期记忆引出的问题

### 问题1：==Context Window 长度限制==

短期记忆装不下怎么办？

**方案：**

- ==**摘要压缩**：让 LLM 把旧对话总结成几句话==
- ==**RAG 化**：把历史对话也向量化，按需检索而不是全部塞入==

---

### 问题2：==记忆的噪声与幻觉==

记忆本身是 LLM 提取和生成的，天然携带幻觉风险

**方案：**

- 记忆写入时加置信度评分
- 定期做记忆审计和去重
- 检索结果加来源标注，让模型自己判断可信度
区分"事实"和"推断"，打上标签。→ 推断类记忆不直接使用，只作参考 → 高重要性记忆使用前让 LLM 再验证一次

---

### 问题3：==记忆检索的相关性问题==

==向量相似度≠语义真正相关==，可能检索回来的记忆根本没用，反而干扰推理。

**方案：**

- **混合检索**：向量检索 + BM25 关键词检索结合
向量检索强在语义理解，BM25（TFIDF升级版）强在关键词匹配
- **Re-ranking**：用 Cross-Encoder 对召回结果重排序
- **检索后过滤**：让 LLM 判断召回记忆是否真正相关

---

### 问题4：==记忆的一致性问题==

用户的偏好、状态会随时间变化，旧记忆可能已经过时。

```
记忆："用户喜欢Python"  ← 三个月前存的
现实：用户已经转Go了
```

**方案：**

- 记忆存入时打时间戳
- 检索时按时间衰减加权
- 新信息和旧记忆冲突时，优先新信息并更新记忆

---

### 问题5：记忆的隐私与安全

多用户场景下，记忆隔离必须严格，否则会有数据泄露风险。

**方案：**

- 记忆按用户 ID 严格隔离
- 敏感信息加密存储
- 访问控制 + 审计日志

---

### 问题6：记忆爆炸（Memory Bloat）

长期运行后记忆库越来越大，检索变慢，噪声增多。
”噪声比信号增长更快“
**方案：**
阶段一：写入时就控制质量（预防）
这是最根本的解法，**垃圾不入库**比入库后再清理效率高得多。

去重检测

信息提炼写入（不存原文）

阶段二：
聚类合并
对于已有记忆进行定期的整理，把相似的记忆合并

阶段三：
遗忘
不同类型记忆的半衰期配置
时效性+重要性+访问频率+置信度 加权综合


冷热分层

## 三、Agent 任务分解怎么实现

### 为什么需要任务分解

复杂任务直接扔给 Agent 会：

- 超出单次推理能力边界
- 无法并行，效率低
- 出错时难以定位和恢复

任务分解的目标是把一个**大而模糊的目标**变成**小而具体的可执行步骤**。

---

### 方法一：LLM 直接分解（最常用）

直接 prompt 让模型输出结构化任务树：

```
Prompt:
"你是一个任务规划专家。请将以下任务分解为
可独立执行的子任务，输出 JSON 格式：

任务：'帮我分析竞品并生成报告'

输出格式：
{
  'tasks': [
    {'id': 1, 'desc': '...', 'depends_on': []},
    {'id': 2, 'desc': '...', 'depends_on': [1]},
    ...
  ]
}"
```

输出示例：

json

```json
{
  "tasks": [
    {"id": 1, "desc": "确定竞品列表", "depends_on": []},
    {"id": 2, "desc": "抓取竞品官网信息", "depends_on": [1]},
    {"id": 3, "desc": "分析各竞品核心功能", "depends_on": [2]},
    {"id": 4, "desc": "生成对比分析报告", "depends_on": [3]}
  ]
}
```

---

### 方法二：层级分解（Hierarchical）

对于特别复杂的任务，做**多层分解**：

```
总目标：开发一个电商网站
   ├── 子任务1：前端开发
   │     ├── 子子任务1.1：搭建页面框架
   │     ├── 子子任务1.2：实现商品列表
   │     └── 子子任务1.3：实现购物车
   ├── 子任务2：后端开发
   │     ├── 子子任务2.1：设计数据库
   │     └── 子子任务2.2：实现API
   └── 子任务3：测试部署
```

实现上可以**递归调用分解 prompt**，直到每个叶子节点足够原子化。

---

### 方法三：依赖图 + 并行执行

分解完任务后，识别依赖关系，**无依赖的任务并行执行**：

```
Task1 ──────────────────→ Task4
Task2 ──┐                  ↑
        ├──→ Task3 ────────┘
Task2 ──┘

Task1 和 Task2 无依赖 → 并行执行
Task3 依赖 Task1、Task2 → 等待后执行
Task4 依赖 Task3 → 最后执行
```

工程实现用 **DAG（有向无环图）**：

python

```python
# 伪代码
dag = build_dag(tasks)  # 构建依赖图
ready_queue = dag.get_ready_tasks()  # 获取无依赖任务

while not dag.is_complete():
    parallel_execute(ready_queue)  # 并行执行
    dag.mark_complete(done_tasks)
    ready_queue = dag.get_ready_tasks()  # 解锁新任务
```

---

### 方法四：动态分解（边执行边分解）

不预先生成完整计划，而是**执行一步再决定下一步**，更灵活：

```
目标 → 分解出 Step1
         ↓ 执行 Step1
       根据结果分解 Step2
         ↓ 执行 Step2
       根据结果分解 Step3
         ↓ ...
```

适合：任务高度动态，前一步的结果决定后续走向。

这本质上就是 **ReAct 的 Thought 层做任务分解**。





## Embedding的原理
把文本（或图片、音频等）转换成高维空间中的一个**数值向量**，语义相近的内容在这个空间里距离也近。检索时，把 query 也转成向量，找最近邻即可。

**余弦相似度**cos(θ) = A·B / (‖A‖‖B‖)最常用，忽略向量长度

ANN 索引算法（为什么不直接暴力搜）
百万级向量暴力计算太慢，实际用近似最近邻（ANN）



# 保证检索准确性的关键手段
- **Chunking 策略** — 块切得好不好是基础，语义完整 + 适当 overlap，后面做再多也补不回来
- **Hybrid Search（混合检索）** — 向量（语义）+ BM25（关键词）融合，互补短板，单靠向量检索准确率普遍不够
- **Reranker 精排** — 粗召回 TopK 后用 Cross-Encoder 重排，是提升精度最直接的手段，效果立竿见影
- **Embedding 模型选型** — 模型要和领域匹配，通用模型在垂直领域掉点很明显



# function calling如何解析用户的意图
### 核心机制

你把"工具清单"（函数名、参数、描述）告诉模型，模型根据用户输入决定**调不调用、调哪个、传什么参数**，本质上是让 LLM 做意图分类 + 实体提取。

```
用户说话 → LLM 理解意图 → 输出结构化 JSON → 你的代码执行函数 → 结果返回
```

---

### 完整流程

```
1. 定义工具
   你告诉模型：我有 get_weather(city, date) 这个函数

2. 用户输入
   "明天北京天气怎么样？"

3. 模型判断
   → 意图：查天气
   → 实体：city=北京, date=明天(tomorrow)
   → 输出：{ "name": "get_weather", "arguments": {"city": "北京", "date": "tomorrow"} }

4. 你的代码执行
   调用真实 API，拿到结果

5. 结果回传给模型
   模型组织成自然语言回复用户
```

---

### 意图解析的关键：工具描述要写好

==模型完全靠**函数描述**来判断意图，描述写得烂就会调错或不调==

## ==切割的依据==

### 1. 按结构切（优先）

文档本身有结构就按结构切，最自然：

```
Markdown  → 按 # ## ### 标题切
PDF/Word  → 按章节、段落切
代码文件  → 按函数/类切
HTML      → 按 <section> <article> 切
```

结构切割能保证每个块语义完整，是最推荐的方式。

### 2. 按语义切

没有明显结构时，按句子边界、段落边界切，不要切断一个完整的意思：

```
❌ 在句子中间切：
"向量数据库通过计算余弦相似度来找到语义相" | "近的内容，这是RAG的核心"

✓ 在句子边界切：
"向量数据库通过计算余弦相似度来找到语义相近的内容。" | "这是RAG的核心原理。"
```

### 3. 按 Token 数兜底

上面两种都没法用时，按固定 token 数切，**加 overlap 防止语义断裂**：

python

```python
chunk_size = 512      # 每块大小
chunk_overlap = 128   # 相邻块重叠部分

# 块1: token 0~512
# 块2: token 384~896   ← 重叠了 128 个 token
# 块3: token 768~1280
```

overlap 的作用是防止一个完整概念被切成两半后两边都不完整。

---

## 块大小怎么选

|场景|建议大小|原因|
|---|---|---|
|问答/FAQ|128~256 token|答案本身就短，小块精准|
|技术文档|512 token|概念需要一定上下文|
|长篇报告|1024 token|保留段落完整性|

块太小：语义不完整，检索出来缺上下文  
块太大：噪声多，相似度被稀释，检索不准

---

## ==父子块技巧（实际项目常用）==

```
父块（1024 token）→ 存原文，用于最终喂给 LLM
    ↓ 切分
子块（256 token） → 存向量，用于检索

检索时：用子块找，返回时：返回父块
```

好处是检索精准（小块），上下文完整（大块）。

---

## 一句话原则

> 切割的核心依据是**语义完整性**，一个块应该能独立回答一个问题，不依赖上下文也能看懂。






## 非阻塞式响应（流式输出）

### 原理

普通请求是等模型生成完再一次性返回，流式是模型**边生成边推送**，用的是 SSE（Server-Sent Events）协议：

```
普通：[等待5秒] → 一次性收到完整回复
流式：token1 → token2 → token3 → ... 逐字输出
```

### 协议层：SSE

本质是 HTTP 长连接，服务端不关闭连接，持续往里写数据，浏览器原生支持，不需要 WebSocket。

响应头设置一个关键字段：

```
Content-Type: text/event-stream
```

告诉浏览器"这个连接会持续推数据，别关"。

---

### 后端：边生成边写

调用模型时开启 stream 模式，模型每生成一个 token 就回调一次，后端拿到就立刻往响应里写：

python

```python
# 关键：用 stream() 而不是 create()
with client.messages.stream(...) as stream:
    for text in stream.text_stream:
        yield text  # 每个 token 立即推出去，不等全部生成完
```

`yield` 是关键，把函数变成生成器，每次产出一个 token 就推给客户端。

---

### 前端：逐块读取

浏览器用 `ReadableStream` 接收，响应体是一个流，不断从里面读：

javascript

# agent需要同时调用多个工具的时候，如何解决工具之间的依赖关系
核心思路是**把工具调用建模成 DAG（有向无环图）**，显式声明依赖关系，然后按拓扑序执行。
**实现层面**，让每个工具声明 `inputs` 和 `outputs` 的 schema，调度器在执行前做依赖分析，把无依赖的工具分配到同一个并发批次，依赖满足后再触发下一批。这就是经典的"波次执行"（wave execution）模式。

## 二、并发冲突

并发主要带来三类问题：

**1. 写写冲突（Write-Write Conflict）**

两个工具同时修改同一资源，后者覆盖前者。解决方案是乐观锁（CAS + 版本号）或悲观锁（执行前获取锁）。
在 agent 场景里，更常见的做法是**工具粒度加互斥**：对同一资源的写操作统一串行化，不让两个写工具同时运行。

**2. 读写不一致（Read-Write Hazard）**

工具 A 读数据，工具 B 同时在写。A 可能读到脏数据。解法是读快照隔离（Snapshot Isolation）——工具启动前对状态打快照，在快照上读，写操作走单独的事务提交。

**3. 速率限制冲突（Rate Limit）**

多个工具并发调同一个外部 API，触发限流。需要在工具调度层维护一个 rate limiter，按 token bucket 或滑动窗口控制并发数。实践中常见的做法是给每个外部服务维护一个信号量（semaphore），并发数超限就排队等待。









### 乐观锁是什么

一种并发控制思路，核心假设是"冲突很少发生"，所以**不真正加锁**，而是在提交时检查数据有没有被别人改过，改过了就回滚重试。

和悲观锁对比理解：

- **悲观锁**：先锁住资源，操作完再释放，别人等着。像厕所占坑，进去就锁门
- **乐观锁**：不锁，操作完提交时检查，有冲突就重来。像 Git 提交，push 前先 pull，有冲突就解决

---

### 锁的是什么

不是真的锁，而是锁**数据的版本**。通常用一个 `version` 字段或时间戳来标记：

sql

```sql
-- 读取时记下版本号
SELECT id, balance, version FROM account WHERE id = 1
-- 拿到：balance=1000, version=5

-- 更新时带上版本号做条件
UPDATE account SET balance=900, version=6
WHERE id=1 AND version=5  -- ← 如果 version 不是 5，说明被人改过，更新失败
```

`WHERE version=5` 就是"锁"的本体，用版本一致性来保证数据没被并发修改。

---

### 适用场景

**适合用乐观锁**：读多写少、冲突概率低的场景

- 商品详情页点赞数更新
- 用户个人资料编辑
- 文档协同编辑（冲突少时）
- 库存扣减（低并发）

**不适合用乐观锁**：写多、冲突频繁的场景

- 秒杀抢购（大量并发同时写，冲突率极高，重试风暴会打垮系统）
- 银行转账（强一致性要求，悲观锁更安全）

---

### 一句话总结

==乐观锁不是真的锁，是用**版本号**做提交前的一致性校验，适合冲突少的读多写少场景，==冲突多了反而因为大量重试降低性能。
